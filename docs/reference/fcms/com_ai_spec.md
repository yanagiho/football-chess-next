**前提：ゲーム仕様書v8 / 技術要件書v2 確定済み**

## 1. 設計方針

### 1-1. 基本構造

全難易度でルールベース＋Gemmaのハイブリッド構成を採用する。

```
[ルールベースエンジン（安全層）]
  ├── 合法手の生成（移動可能HEX・パス先・シュート先を列挙）
  ├── 違法手の除去（ボード外・移動力超過・コスト上限違反等）
  └── 盤面状態の構造化（Gemmaへの入力データ生成）
        │
        │ 盤面状態 + 合法手リスト（JSON）
        │
[Gemma（判断層）]
  ├── 戦略選択（攻撃/守備/ビルドアップ/カウンター）
  ├── 各コマへの指示選択（合法手リストの中から選択）
  ├── 難易度に応じたプロンプトで判断品質と枚数を制御
  └── 出力：11枚分のアクション指示（JSON）
        │
        │ Gemmaの出力
        │
[ルールベースエンジン（検証層）]
  ├── 合法性の最終チェック（Gemmaの出力が合法手リスト内か）
  ├── 違法な指示 → ルールベースの最善手で置換
  └── 難易度フィルター → 指示送信
```

### 1-2. Gemmaが「全て」を判断する理由

| 旧設計（ルールベース主体） | 新設計（Gemma主体） |
|---|---|
| 戦略選択はif-elseの条件分岐 | Gemmaが盤面全体を見て判断 |
| 理想ポジションは固定テーブル | Gemmaが局面に応じて動的に判断 |
| パターン記憶はマニアック限定 | 全難易度でGemmaの文脈理解が自然に対応 |
| 時代傾向はパラメータ調整 | プロンプトに時代の戦術指示を含めるだけ |
| 更新は手動のコード変更 | ファインチューニングで自動改善 |

### 1-3. ルールベースが残る理由

Gemmaは判断層だが、安全層は絶対に必要。

| 安全層の役割 | なぜ必要か |
|---|---|
| 合法手の生成 | HEX距離計算、ZOC判定、パス射程計算は数学的に正確でなければならない。LLMの近似計算では不十分 |
| 違法手の除去 | Gemmaが「移動力4のコマを6HEX動かす」指示を出しても、安全層が弾く |
| 最終検証 | JSON出力のパース失敗、不正なpiece_id、重複指示等をキャッチ |
| フォールバック | Workers AI障害時に自動でルールベース最善手に切り替え。プレイヤーに影響ゼロ |

## 3. 初期学習データの生成

サービスイン時点ではプレイヤーの試合ログが存在しないため、以下の手順で初期学習データを生成する。

### 3-1. ブートストラップパイプライン

```
[Phase 1: ルールベースAI同士の対戦]
  ├── v1のルールベースAI（§4〜§10に記載）同士で10,000試合を自動生成
  ├── 盤面状態→指示のペアを抽出
  └── 初期学習データセット（約90万ターン分）を生成
        │
[Phase 2: Gemma初期ファインチューニング]
  ├── Phase 1のデータでLoRAファインチューニング
  └── 「ルールベースと同等の判断ができるGemma」を生成
        │
[Phase 3: Gemma vs ルールベースの対戦]
  ├── Phase 2のGemmaとルールベースAIで5,000試合
  ├── Gemmaが勝った試合の指示データを「良い手」として抽出
  └── 追加学習データとして蓄積
        │
[Phase 4: Gemma自己対戦]
  ├── Phase 3で改善したGemma同士で5,000試合
  ├── 高スコア試合の指示データを抽出
  └── 最終ファインチューニング
        │
[Phase 5: 品質検証]
  ├── 最終Gemma vs ルールベース（マニアック設定）で1,000試合
  │   → Gemma勝率60%以上で合格
  ├── 各難易度プロンプトでの出力品質チェック
  │   → 合法手出力率95%以上で合格
  └── 処理時間チェック
      → Workers AI上で300ms以内で合格
```

### 3-2. 開発スケジュールへの影響

| タスク | 所要期間 | 前提条件 |
|---|---|---|
| ルールベースAIの実装 | 2週間 | ゲームエンジン完成後 |
| Phase 1: 自動対戦10,000試合 | **2〜3日**（並列100セッションで約30時間。1試合90ターン×50ms/ターン＝4.5秒/試合。直列なら12.5時間だが安全マージン含む） | ルールベースAI完成後 |
| Phase 2: 初期ファインチューニング | 2〜3日（GPU学習） | Phase 1完了後 |
| Phase 3: Gemma vs ルールベース5,000試合 | 1〜2日（並列50セッション。Gemma推論300ms/ターン含め1試合約30秒） | Phase 2完了後 |
| Phase 4: 自己対戦5,000試合 | 1〜2日（同上） | Phase 3完了後 |
| Phase 5: 品質検証 | 1日 | Phase 4完了後 |
| **合計** | **約3〜4週間** | ゲームエンジン完成後に開始 |

> Phase 1のルールベース同士の対戦はGemma推論が不要なため高速（50ms/ターン）。Phase 3以降はGemma推論が入るため1ターン300ms程度に低下する。並列度はCloudflare Workers/DOの同時実行数制限内で調整。

## 4. 局面評価（ルールベース安全層）

ルールベースエンジンが盤面を数値化し、Gemmaに構造化データとして渡す。

### 4-1. 評価関数

```
盤面スコア = ボール位置スコア
           + コマ配置スコア
           + ZOC支配スコア
           + 得点差スコア
           + 残りターンスコア
```

### 4-2. ボール位置スコア

| ボール位置 | 自チーム保持時 | 相手チーム保持時 |
|---|---|---|
| ファイナルサード | +50 | −50 |
| アタッキングサード | +30 | −30 |
| ミドルサードA | +10 | −10 |
| ミドルサードD | −10 | +10 |
| ディフェンシブサード | −30 | +30 |
| ディフェンシブGサード | −50 | +50 |

### 4-3. コマ配置スコア

| 状態 | スコア |
|---|---|
| コマが自分のゾーンボーナス対象ゾーンにいる | +5 / 枚 |
| FW / WG / OM がアタッキングサード以上 | +8 / 枚 |
| DF / SB / VO がディフェンシブサード以下 | +5 / 枚 |
| GK が PA内 | +20 |
| GK が PA外 | −30 |

### 4-4. ZOC支配スコア

```
ZOC支配率 = 自チームのZOCが覆うミドルサードHEX数 / ミドルサード全HEX数
ZOC支配スコア = (ZOC支配率 − 0.5) × 40
```

### 4-5. 得点差・残りターンスコア

| 状態 | 効果 |
|---|---|
| リード | 守備ボーナス+20 |
| 同点 | 中立 |
| ビハインド | 攻撃ボーナス+20 |
| 残り10ターン以下でビハインド | 追加攻撃+20 |
| 残り5ターン以下でリード | 追加守備+20 |

## 5. 合法手生成（ルールベース安全層）

### 5-1. 各コマの合法手リスト

ルールベースエンジンが毎ターン全11枚の合法手を列挙し、Gemmaに渡す。

| コマの状態 | 合法手の種類 |
|---|---|
| ボール非保持 | 移動（ZOCで停止するHEX含む）、交代 |
| ボール保持 | 移動、ドリブル、パス（射程内の全味方）、シュート（射程内ならゴール6ゾーン）、交代 |
| 交代不可（枠切れ or コスト超過） | 交代を除外 |

### 5-2. Gemmaへの合法手データ形式

```json
{
  "piece_id": "p09",
  "position": "OM",
  "cost": 3,
  "current_hex": [14, 8],
  "has_ball": true,
  "legal_actions": [
    {"id": "a1", "action": "dribble", "target": [15, 7], "note": "前方、ZOC外"},
    {"id": "a2", "action": "dribble", "target": [15, 9], "note": "前方、相手DF ZOC内→停止+タックル"},
    {"id": "a3", "action": "pass", "target_piece": "p11", "note": "FW★2.5、ファイナルサード、パスカット推定20%（※移動前盤面基準）"},
    {"id": "a4", "action": "pass", "target_piece": "p10", "note": "WG★1.5、右サイド、パスカット推定5%（※移動前盤面基準）"},
    {"id": "a5", "action": "shoot", "zone": "top_left", "note": "距離5HEX、ブロック推定30%、成功推定85%"},
    {"id": "a6", "action": "shoot", "zone": "bottom_right", "note": "距離5HEX、ブロック推定10%、成功推定85%"}
  ]
}
```

> **重要：パス関連の確率は移動前盤面での推定値。** 実際のパスはフェーズ2（移動後の盤面）で解決されるため、守備コマの移動によって実際の確率は変動する。Gemmaにはこの旨をシステムプロンプトに明記する：「パスカット確率は移動前の推定値です。相手コマが移動してパスコースから外れる可能性、またはパスコースに移動してくる可能性があります。相手の移動を予測した上で判断してください。」

## 7. 相手プレイヤー傾向の分析

マニアック難易度で使用する、試合中の相手プレイヤーの傾向分析。

### 7-1. 蓄積する情報

| プレス強度 | 守備時にアタッキングサードまでコマを上げる枚数 | 「相手は守備時に3枚をアタッキングサードに残す」 |

## 9. 技術仕様

### 9-1. Gemma実行環境

| 項目 | 仕様 |
|---|---|
| モデル | Gemma 3（Cloudflare Workers AI対応版） |
| モデル指定方式 | **環境変数（Workers Secrets）で管理**。コード変更なしにモデル入替可能 |
| 実行環境 | Cloudflare Workers AI（エッジ推論） |
| 入力 | システムプロンプト＋盤面状態＋合法手リスト（JSON） |
| 出力 | 11枚分のアクション指示（JSON） |

### 9-3. Gemma出力のパースと検証

```typescript
function parseGemmaOutput(raw: string, legalMoves: LegalMoves): Order[] {
  // 1. JSONパース
  const parsed = JSON.parse(raw);
  
  // 2. 各orderを検証
  const validOrders = parsed.orders.filter(order => {
    // piece_idが自チームのフィールドコマか
    if (!isOwnFieldPiece(order.piece_id)) return false;
    // 同一piece_idの重複チェック
    if (isDuplicate(order.piece_id, validOrders)) return false;
    // actionが合法手リスト内に存在するか
    if (!isInLegalMoves(order, legalMoves)) return false;
    // 交代のコスト上限チェック
    if (order.action === 'substitute' && !checkCostLimit(order)) return false;
    return true;
  });
  
  // 3. 不正なorderはルールベース最善手で置換
  const finalOrders = fillWithRuleBasedFallback(validOrders, legalMoves);
  
  return finalOrders;
}
```

### 9-4. フォールバック戦略

| 障害パターン | 対応 |
|---|---|
| Workers AI応答遅延（500ms超過） | ルールベースの最善手を全11枚分出力 |
| JSON パースエラー | 同上 |
| 合法手外の指示が過半数 | 全指示をルールベースで置換＋エラーログ記録 |
| Workers AI完全障害 | ルールベースのみで全試合を処理。プレイヤーに影響なし |
