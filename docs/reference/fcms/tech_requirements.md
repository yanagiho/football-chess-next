## 3. 通信設計

### 3-1. 通信モデル

| フェーズ | プロトコル | 認証 |
|---|---|---|
| マッチング | WebSocket (WSS) | 接続時にトークン検証（§7-2） |
| ターン入出力 | WebSocket (WSS) | セッション内検証 |
| ミニゲーム | WebSocket (WSS) | セッション内検証 |
| 非対戦系API | REST (HTTPS) | Bearerトークン |

### 3-2. ターン入力データ構造

```json
{
  "match_id": "m_abc123",
  "turn": 42,
  "player_id": "p_xyz789",
  "sequence": 84,
  "nonce": "a1b2c3d4e5f6",
  "orders": [
    { "piece_id": "p01", "action": "move", "target_hex": [12, 8] },
    { "piece_id": "p05", "action": "pass", "target_piece": "p09" },
    { "piece_id": "p11", "action": "substitute", "bench_piece": "b03" }
  ],
  "client_hash": "sha256_of_orders",
  "timestamp": 1720000000000
}
```

| フィールド | 目的 |
|---|---|
| sequence | 単調増加のシーケンス番号。リプレイ攻撃防止。過去のsequenceは拒否 |
| nonce | ワンタイム一意値。同一nonceの再送信を拒否 |
| client_hash | ordersのSHA-256ハッシュ。通信途中の改ざん検知 |

### 3-3. レイテンシ要件

| 要件 | 目標値 |
|---|---|
| ターン入力→解決結果配信 | 2秒以内 |
| マッチング成立 | 30秒以内（超過でCOM提案） |
| REST API応答 | 500ms以内 |

## 4. グローバル対戦インフラ

### 4-1. Cloudflareエッジネットワーク

Workers / Durable Objectsは300+拠点で自動配置。リージョン管理不要。

### 4-2. マッチメイキング（シャード構成）

単一Durable Objectのボトルネックを回避するため、リージョンごとにシャードを分割する。

```
[Matchmaking Shard: Asia]     ─── 東アジア・東南アジア・オセアニア
[Matchmaking Shard: Americas] ─── 北米・南米
[Matchmaking Shard: Europe]   ─── ヨーロッパ・中東・アフリカ
[Matchmaking Coordinator (Primary)]   ─── クロスリージョンマッチング
[Matchmaking Coordinator (Standby)]   ─── Primary障害時に自動昇格
```

> Coordinatorは2つのDurable Objectで冗長化。Primaryが10秒以上応答しない場合、各シャードはStandbyに切り替える。Standbyは直近のキュー状態をKVからリストアして処理を継続。

| フェーズ | 処理 |
|---|---|
| 0〜10秒 | 自リージョンシャード内で±200レーティング検索 |
| 10〜20秒 | 条件を±400に緩和 |
| 20〜30秒 | Coordinatorが全シャードに照会。クロスリージョンマッチング |
| 30秒超 | COM対戦を提案 |

> 各シャードは独立したDurable Object。1シャードあたり最大数百の同時待ちプレイヤーを処理。シャード内がさらに混雑する場合はサブシャードに動的分割。

### 4-3. Durable Objectによるゲームセッション

```
[Durable Object: GameSession_m_abc123]
├── WebSocket接続: Player A（認証済み）
├── WebSocket接続: Player B（認証済み）
├── ゲーム状態（ボード・スコア・ターン数）
├── ターン入力バッファ（sequence/nonce検証付き）
├── 判定エンジン（§7の全判定式）
├── ターンタイマー（1分）
├── 切断監視タイマー（30秒猶予）
└── 試合ログ（R2永続化キュー経由）
```

### 4-4. WebSocket Hibernation対応

Durable ObjectsのWebSocket Hibernation APIを使用し、アイドル時のリソース消費を最小化する。

| 状態 | 処理 |
|---|---|
| ターン入力待ち | Hibernation状態。メモリ消費最小。WebSocketメッセージ受信で自動復帰 |
| ターン解決中 | アクティブ状態。判定処理を実行 |
| ミニゲーム中 | アクティブ状態 |
| 切断猶予中 | アクティブ状態。30秒タイマー監視 |

> ゲーム状態はDurable Objectの永続ストレージ（transactional storage）に書き込むため、hibernate/復帰でも状態が失われない。

| チーム編成 | Cloudflare D1 | ゲーム |
| 試合サマリ | Cloudflare D1 | ゲーム |
| 試合詳細ログ（全ターン） | Cloudflare R2 (JSON) | ゲーム |
| ランキング | Cloudflare KV | ゲーム |
| 進行中ゲーム状態 | DO transactional storage | ゲーム |

### 5-2. D1の制約対策

D1はSQLiteベース（最大10GBだが書き込み並行性が低い）のため、以下の運用ルールを適用。

| 対策 | 内容 |
|---|---|
| 書き込みの非同期化 | 試合結果はDurable Object → Cloudflare Queues → D1の非同期パイプラインで書き込み。試合終了の同時集中を緩衝 |
| 詳細ログの分離 | 全ターンログ（1試合300KB〜）はD1に入れず、R2にJSON保存。D1には試合サマリ（結果・スコア・日時のみ）を格納 |
| テーブル分割 | 試合サマリは月次テーブル分割（matches_2025_07等）。古い月のデータはR2にアーカイブ |
| サイズ監視 | D1のサイズを定期監視。閾値（5GB）超過でアラート→アーカイブ実行 |

### 5-3. データ量の見積もり

| データ | 1試合あたり | 1万DAU・月間想定 |
|---|---|---|
| 試合サマリ（D1） | 約0.5KB | 約750MB/月（1日5万試合想定） |
| 詳細ログ（R2） | 約300KB（圧縮後） | 約450GB/月 |

## 6. フロントエンド設計

### 6-1. HEXボードUI

| 項目 | 仕様 |
|---|---|
| 描画方式 | **レイヤー分離構成（PixiJS不使用）** |
| 背景レイヤー | 芝生＋HEXグリッド＋白線＋ゴールを1枚の画像として事前描画（画像ソフトで制作） |
| HEX座標マップ | 全748マス（22×34）の中心座標(x,y)をJSONで事前計算。全レイヤーがこの座標を参照 |
| オーバーレイレイヤー | HTML Canvas。ZOCハイライト・移動範囲・パスライン・シュートラインを座標マップに基づいて描画 |
| コマレイヤー | DOM要素（`<img>` 絶対配置）。スプライト画像を座標マップに従って配置・移動 |
| UIレイヤー | HTML/CSS。ボタン・タイマー・スコア等の通常WebUI |
| ズーム・パン | CSS transform（scale/translate）でボード全体を変換。ピンチ/ホイール/ドラッグ対応 |
| FPS | 30fps以上（コマ移動アニメーション時） |

> **設計意図**：PixiJSを使わないことでバンドルサイズが大幅に削減され、初回ロード時間が短縮される。HEXの描画をコードで計算する必要がなく、画像ソフトで描いた背景の上にDOM要素を置くだけのシンプルな構成。Claude Codeでの生成精度も高くなる。

## 7. セキュリティ

### 7-1. 通信セキュリティ

| 対策 | 内容 |
|---|---|
| TLS | 全通信をHTTPS / WSS。Cloudflareが自動適用 |
| CORS | ゲームのオリジン（`https://manics.example.com`）のみ許可。ワイルドカード禁止 |
| CSP | `default-src 'self'; script-src 'self'; connect-src 'self' wss://manics.example.com https://api.football-century.example.com; img-src 'self' https://r2.example.com;` |
| HSTS | `Strict-Transport-Security: max-age=31536000; includeSubDomains` |
| DDoS | Cloudflare標準DDoS対策（全レイヤー自動適用） |

### 7-2. WebSocket認証

```
1. クライアントがプラットフォームから認証トークン（JWT）を取得
2. WebSocket接続時、URLクエリパラメータにトークンを含める
   wss://game.example.com/match?token=eyJhbG...
3. WorkerのWebSocket upgradeハンドラでトークンを即時検証：
   a. 署名検証（プラットフォームの公開鍵で）
   b. 有効期限チェック（試合全体をカバーする残存期間が必要：最低2時間）
   c. ユーザーIDとマッチングの対戦者IDの一致確認
4. 検証失敗 → upgradeを拒否（HTTP 401）。WebSocket接続自体が確立されない
5. 検証成功 → WebSocket接続確立。以降のメッセージはセッション内で認証済み
```

> 旧設計の「接続後に最初のメッセージでトークン送信」は、接続確立〜認証完了の間に未認証ウィンドウが生じるため廃止。upgradeハンドラで検証することで、未認証接続が一切存在しない。

### 7-3. サーバーサイド入力バリデーション（完全リスト）

Durable Object内で全ターン入力に対して以下を検証。1つでも違反があればその指示を無視（不正ログに記録）。

| # | チェック項目 | 違反時の処理 |
|---|---|---|
| 1 | player_idが当該試合の参加者か | 全指示無視 |
| 2 | sequenceが前回+1の単調増加か | 全指示無視（リプレイ攻撃） |
| 3 | nonceが未使用か | 全指示無視（リプレイ攻撃） |
| 4 | timestampが現在時刻±5秒以内か | 全指示無視 |
| 5 | ordersの件数が11以下か | 超過分を切り捨て |
| 6 | 各piece_idが自チームのフィールドコマか | 該当指示を無視 |
| 7 | 同一piece_idが重複していないか | 2件目以降を無視 |
| 8 | actionが許可された値か（move/pass/shoot/dribble/substitute/skill） | 該当指示を無視 |
| 9 | target_hexがボード範囲内（0-21, 0-33）か | 該当指示を無視 |
| 10 | 移動距離がコマの移動力（+ゾーンボーナス+コスト修正）以内か | 該当指示を無視 |
| 11 | パス/シュートのtargetが有効か | 該当指示を無視 |
| 12 | 交代後のフィールド総コストが16以下か | 交代指示を無視 |
| 13 | 交代回数が残り枠以内か | 交代指示を無視 |
| 14 | ボール保持コマ以外がpass/shoot/dribbleしていないか | 該当指示を無視 |

### 7-4. レート制限

| 対象 | 制限 | 超過時 |
|---|---|---|
| WebSocketメッセージ | 1接続あたり10msg/秒 | 超過メッセージ破棄。3回連続超過で警告 |
| ターン入力 | 1ターンにつき1回（再送は3回まで） | 超過を無視 |
| REST API（全体） | 1ユーザーあたり60req/分 | 429 Too Many Requests |
| マッチングリクエスト | 1ユーザーあたり5req/分 | 429返却 |
| ショップAPI | 1ユーザーあたり10req/分 | 429返却 |

### 7-5. プラットフォームAPI間通信の保護

| 対策 | 内容 |
|---|---|
| 認証 | ゲーム→プラットフォームのAPI呼び出しにサービスAPIキー（Workers Secrets格納） |
| 署名検証 | プラットフォームのレスポンスにHMAC-SHA256署名。ゲーム側で検証 |
| 相互TLS | 将来的にmTLSを検討（Cloudflare mTLS対応） |
| レスポンス検証 | 所持コマ数が200超でないか、コストが1〜3の範囲内かなど、ビジネスロジック整合性チェック |

### 7-6. 不正対策

| 脅威 | 対策 |
|---|---|
| 入力改ざん | §7-3の完全バリデーション。ゲームロジック100%サーバーサイド |
| リプレイ攻撃 | sequence + nonce の二重チェック |
| BOT | Cloudflare Turnstile（マッチング前）。入力パターン分析（全ターン即送信 or 常にギリギリの異常パターン検知） |
| マルチアカウント | プラットフォーム側の認証強度に依存。ゲーム側では短期間に同一IP・類似デバイスから複数アカウントの対戦がある場合にフラグ。ランクマッチではフラグ付きアカウントの勝利レーティング付与を保留 |
| XSS | CSPヘッダー（§7-1）で外部スクリプト実行を遮断 |
| CSRF | SameSite=Strict Cookie + カスタムヘッダー検証 |

## 8. 切断・異常系処理

### 8-1. プレイヤー切断

| フェーズ | 処理 |
|---|---|
| 切断検知 | WebSocket close / pingタイムアウト（10秒無応答）で切断と判定 |
| 猶予期間 | **30秒間**の再接続猶予。相手には「対戦相手が切断しました。再接続を待っています...」と表示 |
| 猶予中のターン | 切断中のプレイヤーは全コマ「指示なし」（静止）として処理。試合は進行する |
| 再接続成功 | Durable Objectの同じゲームセッションに再接続。現在のゲーム状態を送信して試合続行 |
| 再接続失敗（30秒超過） | **自動敗北**。相手に勝利判定。ランクマッチの場合はレーティング反映 |
| 両者同時切断 | 両者に30秒猶予。先に再接続した方が続行。両者タイムアウトの場合は引き分け |

### 8-2. プラットフォームAPI障害時のフォールバック

| 状況 | 処理 |
|---|---|
| 起動時にAPIが応答しない | キャッシュ（KV）に保存された直近の所持コマ情報を使用。「一部情報が古い可能性があります」と表示。COM対戦のみ許可 |
| 試合中にAPIが落ちる | **影響なし**。試合に必要な情報は試合開始時にDurable Object内に保持済み。試合は正常に完了する |
| 試合後にAPIが落ちる | 試合結果のレーティング更新はゲーム側D1に保存。プラットフォームへの反映はQueues経由でリトライ |
| ショップが使えない | 「現在メンテナンス中です」表示。対戦機能には影響なし |

**キャッシュ戦略：**
- 所持コマ情報：KVに保存（TTL 1時間）。API呼び出し前にキャッシュ確認
- コママスタ：D1に保存（更新時のみ同期）。プラットフォーム障害と無関係
- セーブ権：KVに保存（TTL 24時間）。障害時は直近の状態を信頼

**キャッシュ即時無効化：** プラットフォームからのWebhookでKVキャッシュを即時削除する。
- コマ購入完了時：プラットフォーム → `POST /webhook/purchase` → 該当ユーザーの所持コマキャッシュを削除
- Webhookは署名検証（HMAC-SHA256）付き。署名不一致のリクエストは拒否
- Webhook障害時：キャッシュTTL（1時間）の自然失効で最終的に整合する

### 8-3. Durable Object障害

| 状況 | 処理 |
|---|---|
| DO内部エラー | try-catchで捕捉。エラーログをR2に記録。可能なら試合続行、不可なら両者に「サーバーエラーにより試合が中断されました」→引き分け扱い |
| Cloudflare基盤障害 | DOのtransactional storageにゲーム状態が永続化済みのため、基盤復旧後に同一DOが復活し試合再開可能 |
| 長時間基盤障害 | 5分以上復旧しない場合、該当試合は無効試合（ランクマッチの場合はレーティング変動なし） |

## 10. バックアップ・災害復旧

### 10-1. バックアップ

| データ | 方式 | 頻度 |
|---|---|---|
| D1 | Cloudflare D1の自動バックアップ（Point-in-time recovery） | 継続的 |
| R2 | R2の標準耐久性（99.999999999%）で保証。追加バックアップは不要 |  |
| KV | 揮発性キャッシュのため不要。再構築可能なデータのみ格納 | — |
| Durable Object storage | DOの永続ストレージは自動レプリケーション | 継続的 |

### 10-2. 災害復旧計画

| シナリオ | RTO（目標復旧時間） | RPO（目標復旧時点） | 手順 |
|---|---|---|---|
| D1障害 | 1時間 | 直前 | Point-in-time recoveryで復元 |
| R2障害 | Cloudflareのインフラ復旧に依存 | データ損失なし | R2の冗長性で自動復旧 |
| 単一DO障害 | 即時 | 直前のターン | transactional storageから自動復旧 |
| Cloudflare全面障害 | Cloudflareの復旧に依存 | 直前 | 進行中試合は中断→復旧後に再開 or 無効試合 |
