# Unity版 FootballChess サッカールール調査

調査日: 2026-06-23  
対象リポジトリ: `GadeDev/football-chess-app`

## 前提となる盤面モデル

Unity版は FCMS の 22x34 HEX 盤面ではない。5列程度の抽象サッカー盤で、セル座標 `CellPosition(x, y)` と `CellType` によってエリアを表現している。

主なエリアは次の通り。

| エリア | 意味 |
| --- | --- |
| `Goal` | ゴールセル |
| `GoalArea` | ゴールエリア |
| `PenaltyArea` | ペナルティエリア |
| `VitalArea` | バイタルエリア |
| `CrossArea` | クロスエリア |
| `Normal` | 通常セル |

シュート可能セルは `GoalArea` / `PenaltyArea` / `VitalArea`。`CrossArea` はFK対象や消極的戦術エリアには使われるが、通常シュート可能セルには含まれない。

1セルに入れるコマ数は最大3。同チーム最大数も3で、敵味方混在は許容される。この「同じセルに複数枚いる」状態が、パスカット、タックル、ファウル、シュート補正の中心になっている。

## ターン解決の大枠

`StateBattleCalculate.OnEnterState()` がホスト側でターンを解決する。

処理順は概ね次の通り。

1. ターン開始時のコマ・ボール位置へ表示を戻す。
2. 各プレイヤーの操作を収集し、`LandingPass` の受け手移動を優先するなどして処理順を整える。
3. 操作を順に処理する。
4. ターン中にボールが動いていなければ、ボール保持者と同じセルにいる敵がタックルする。
5. ボールがフリーなら、同じセルの最高コストコマが拾う。
6. フリーボールを最後に触ったチームと同じチームが拾った場合、スルーパス由来のオフサイドを判定する。
7. 遅延行為を判定する。
8. 消極的戦術を判定する。

操作種別ごとの扱いは次の通り。

| 操作 | 解決 |
| --- | --- |
| `Move` | コマだけ移動 |
| `Dribble` | ボール保持コマが移動し、接触相手がいればタックル |
| `Pass` | コマへの通常パス |
| `LandingPass` | 受け手の移動を含むパス |
| `ThroughPass` | セルへのスルーパス |
| `Shoot` | シュート |

## パスとパスカット

通常パスは、通過マス、着地マス、GK特殊処理、オフサイドの順に判定される。

1. パス元からパス先まで `BoardCalculator.GetRoute()` で通過セルを作る。
2. 各通過セルで `FlyingPass` 成功率を計算し、`100 - 成功率` をパスカット率として抽選する。
3. カットされた場合、そのセルの敵のうち最高コストコマがボールを得る。
4. パス先が攻撃側から見た `GoalArea` で、そこに相手GKがいれば100%カットされる。
5. 着地セルで `LandingPass` 成功率を計算し、`100 - 成功率` をパスカット率として抽選する。
6. パス先コマに対してオフサイド判定を行う。
7. すべて通ればパス成功。

`LandingPass` は受け手を指定セルへ移動させるパスで、通過カット、着地カット、オフサイドの構造は通常パスと同じ。ターン操作の並びでは、LandingPass の対象コマの移動が優先処理される。

`ThroughPass` はセルへのパスで、直接のオフサイド判定は行われない。ボールがフリーになり、最後に触ったチームと同じチームのコマが拾った時に `IsFreeBallCatchOffside()` で判定される。

### パス成功率の補正

通過マスのパス成功率:

- 通過セルに敵がいなければ100%成功。
- 敵がいる場合、最高コスト敵とパサーのコストから `DrawingFlyingPassMaster` を参照する。
- 通過セルに敵が複数いると成功率 -40。
- パサーがMFなら成功率 +10。
- 消極的戦術ペナルティ中なら成功率が下がる。

着地マスのパス成功率:

- 着地セルに敵がいなければ100%成功。
- スルーパスの着地判定では、受け手コマがないため成功率100%扱いになる。
- 敵がいる場合、最高コスト敵と受け手コマのコストから `DrawingLandingPassMaster` を参照する。
- 着地セルに敵が複数いると成功率 -20。
- 着地セルに受け手以外の味方がいれば、1枚につき成功率 +10。
- パサーMF補正はマスター上0。

注意点:

- `DrawingThroughPassMaster` と `ThroughPass` 用バフは存在するが、確認した主要処理ではスルーパス固有の確率表は使われていない。
- `GetPassSuccessProbability()` 内に「PenaltyArea かつ GoalArea ならGKカット」という条件があるが、単一セルが両方になることはなく、実質的には呼び出し元の GoalArea GK 100%カットが有効な処理になっている。

## シュート、GK、CK

シュート処理は、シュートブロック、シュート成功、失敗後のCK/GKの順に解決される。

1. `shoot` カットインを追加する。
2. シュート地点からゴールセルまでのルートを作る。
3. ルート上にGK以外の敵がいるセルでは、シュートブロック回避率65%で抽選する。
4. ブロック回避に失敗すると、そのセルの最高コスト敵がボールを得てシュートストップ。
5. ブロックを越えたらシュート成功率を抽選する。
6. 成功ならゴール。
7. 失敗ならCKになるかGKになるかを抽選する。
8. CKになった場合、CK成功ならゴール。失敗後は最大3回までCK/GK抽選を繰り返す。
9. CKにならなければGKボール。GKがGoalAreaにいればSaving演出、そうでなければFailedShoot演出。

シュート成功率は相手GKの位置に強く依存する。

- 相手GKがシューターと同じセル、または攻撃側から見たGoalAreaにいる場合のみ、PA/VAシュート確率表を使う。
- それ以外の位置にGKがいる場合、シュート成功率は100%。
- PA/GAからのシュートは `DrawingPenaltyAreaShootMaster`。
- VAからのシュートは `DrawingVitalAreaShootMaster`。
- PA/GAシュートはFW +10、同セル味方1枚ごとに +10、同セル敵1枚ごとに -20。
- VAシュートはFW +5、同セル敵1枚ごとに -25。

CK/GK分岐は `FailedKickType` とGKコストで決まる。主な傾向として、PAシュート失敗後はGKコストが高いほどCKになりにくく、CK失敗後はGKコストが高いほど再CKになりやすい。

## タックルとファウル

タックルは次の場面で発生する。

- ボール保持者がドリブルで敵のいるセルへ入った時。
- 非保持者が移動し、その移動先に敵ボール保持者がいる時。
- ターン中にボールが動かなかった場合、ボール保持者と同セルにいる敵が順にタックルする。

タックル処理では、まずファウルを抽選し、その後に必要ならタックル成功を抽選する。

### ファウル判定

ファウル確率は `sameCellDefencingPieceNum`、つまりタックル側チームの同セルコマ数だけで決まる。攻撃側コマのコスト、守備側コマのコスト、ポジション、エリアはファウル確率そのものには使われない。

ファウル発生後の処理:

| ボール位置 | 処理 |
| --- | --- |
| 攻撃側から見たGoalAreaまたはPenaltyArea | PK |
| 攻撃側から見たVitalAreaまたはCrossArea | FK |
| それ以外 | 通常タックル判定へ進む |

PK/FKのキッカーは、ファウルされたボール保持者そのものではなく、ファウルされた側チームの最高コストコマ。守備側はファウルした側のGK。

PK/FKが失敗した場合は、CK/GK判定へ移る。CKは最大3回まで連鎖する。

注意点:

- 確認したcheckoutでは `DrawingFoulMaster.json` が `Assets/StreamingAssets/MasterData/` に存在しない。コードは `DrawingFoulProbabilityMasterEntry` を読む設計だが、非base64側の型名にも不整合が見えるため、Unity版の実値はこのcheckoutだけでは確定できない。
- Unity版では「ファウルを先に判定し、PA/GA/VA/CrossAreaならイベントキック化」する。FCMS現行仕様の「タックル成功後のファウルでタックルを無効化」とは判定順が違う。

### タックル成功率

タックル成功率は、まず `DrawingTackleMaster` から「ボール保持側の回避率」を取り、補正後に `100 - 回避率` として求める。

主な補正:

- ボール保持者がMFなら回避率 +10。
- タックル側がDFなら回避率 -15。
- ボール保持者のセルに敵が複数いれば回避率 -20。
- ボール保持者のセルに味方がいれば、1枚につき回避率 +10。
- タックル側が消極的戦術ペナルティ中なら回避率が下がる。

タックル成功時はボールがタックル側へ移る。失敗時は `ballKeep` 演出になる。

## オフサイド

Unity版のオフサイドは、現実サッカーの完全再現ではなく、相手守備ラインと移動方向に基づく確率抽選である。

### オフサイドライン

`BoardCalculator.GetOffsideLine(enemyPieces)` が相手コマからラインを決める。

- 相手が複数いれば、原則として2枚目の相手コマのy座標。
- 相手が1枚だけなら、その1枚のy座標。
- ハーフウェイラインを超えないように制限される。
- チーム向きが逆の場合は座標を反転して判定する。

つまり、サッカーでいう「後方から2人目の守備者」を抽象化したラインである。

### 通常パス / LandingPass のオフサイド

通常パスとLandingPassでは、パスカット判定がすべて終わったあとにオフサイド判定を行う。

判定に使う値:

- `offsideLine`: 相手2枚目ライン。
- `beforeLine`: パス元のy座標。
- `afterLine`: パス先のy座標。
- `beforeLine` と `afterLine` の大小関係から、縦移動、横移動、戻り移動を分類する。
- それぞれがオフサイドラインより上か、同ラインか、下かでテーブルを引く。

`DrawingOffsideMaster.json` の確率はおおまかに次の設計になっている。

| 位置関係 | オフサイド率 |
| --- | --- |
| すでにライン裏にいる、またはライン裏へ進む | 100% |
| ライン上に残る、またはラインへ入る | 50%になるケースあり |
| ライン手前に残る、または戻る | 0% |

オフサイドが成立した場合でも、演出上どこで発生したかわかるようにボールは一度パス先へ動かされ、その後 `Offside` コマンドが追加される。

### スルーパスのオフサイド

スルーパスはセルへボールを出すため、パス実行時点では通常の `IsOffside()` を呼ばない。

ターン後半でフリーボールを拾う時に、次の条件で判定される。

- ボールを最後に触ったチームと、拾ったコマのチームが同じ。
- 拾ったコマのターン開始時y座標が、相手2枚目ラインより奥にいる。

このフリーボールキャッチ時のオフサイドは確率抽選ではなく、ライン比較による決定的判定である。

## 遅延行為

同じチームがボールを保持したままターンを終え、かつボールが自陣にある場合に遅延カウントが増える。

- 閾値は3ターン。
- どちらのチームもボールを持っていない場合はカウントリセット。
- ボール保持チームが自陣ではなく敵陣にいる場合もカウントリセット。
- パスカット、シュートストップ、セービング、ゴール、ファウル、タックル成功、オフサイドなどでカウントはリセットされる。

3ターンに達すると `BattleDelay` が発生する。

## 消極的戦術

`IsPassiveTactics()` は「味方9コマ以上が自陣深部に固まっていて、ボールがその自陣深部にない」状態を検出する。

条件:

- ボールがそのチームの `PassiveTacticsArea` にない。
- そのチームの9コマ以上が `PassiveTacticsArea` にいる。

`PassiveTacticsArea` は、自チームから見た下2段相当で、`GoalArea` / `PenaltyArea` / `VitalArea` / `CrossArea` を含む。

発生するとチームがペナルティリストに入り、パスカット回避やタックル回避にデバフが入る設計。ただし、確認したcheckoutでは `DrawingPassiveTacticsMaster.json` が見当たらず、実値は確定できない。

## PK / FK / CK

Unity版のPK/FK/CKは、FCMSのミニゲームではなく、キッカーとGKのコストによる確率抽選で処理される。

| キック | 成功率参照 |
| --- | --- |
| PK | `DrawingPKMaster` |
| FK | `DrawingFKMaster` |
| CK | `DrawingCKMaster` |

失敗後にCKになるかGKになるかは次のマスターで決まる。

| 失敗種別 | 参照マスター |
| --- | --- |
| PA/GAシュート失敗 | `DrawingFailedPenaltyAreaShootMaster` |
| VAシュート失敗 | `DrawingFailedVitalAreaShootMaster` |
| PK失敗 | `DrawingFailedPKMaster` |
| FK失敗 | `DrawingFailedFKMaster` |
| CK失敗 | `DrawingFailedCKMaster` |

CKは最大3回まで連鎖する。CKにならなかった時点でGKボールになる。

## FCMSへ取り込む時の要点

FCMSはHEX盤面、同時指示、ZOC、セットプレーミニゲームを持つため、Unity版をそのまま移植するより「判定思想」を翻訳するのがよい。

優先して参考にできる点:

- オフサイドを「相手2枚目ライン + パス元/パス先の前後関係」で判定する構造。
- スルーパスは出した時点ではなく、味方がスペースで拾った時にオフサイドを見る構造。
- パスを「通過カット」と「着地カット」に分ける構造。
- 同セルまたは同HEX周辺の味方/敵密度で、タックル・パス・シュートへ補正をかける構造。
- 遅延行為と消極的戦術を、サッカーらしい停滞対策として別ルールにする構造。

そのまま持ち込むと危ない点:

- Unity版の盤面は小さな矩形セルで、FCMSのHEX距離・ゾーン・ZOCとは空間粒度が違う。
- Unity版のファウルはタックル前抽選だが、FCMS現行仕様はタックル後ファウル優先の考え方を持っている。
- Unity版のPK/FK/CKは抽選だけで、FCMSのミニゲーム仕様とは別物。
- Unity版のスルーパス固有マスターは主要処理で使われていないように見える。
- `DrawingFoulMaster.json` と `DrawingPassiveTacticsMaster.json` は確認したcheckoutに存在せず、確率実値は別ソース確認が必要。

## FCMS向けの実装案メモ

今後FCMSへ反映するなら、次の形が自然。

1. `src/engine/offside.ts` は「守備側2枚目ライン」「攻撃側半分」「パス元/パス先」「スペースへの到達者」を入力に取る。
2. 通常パスとLanding/スペースパスでオフサイド判定タイミングを分ける。
3. ファウルは現行FCMS仕様に合わせ、タックル成功後のイベントとして整理する。ただしUnity版の「PA/GAならPK、VA/CrossならFK」は採用候補。
4. CK/FK/PKミニゲームは、Unity版の確率抽選をそのまま置き換えるのではなく、成功率・リスク設計の参考値として使う。
5. 消極的戦術と遅延行為は、HEX盤面上の自陣ゾーン枚数・連続保持ターンで再定義する。

## 現行FCMSソースとの比較と置換判断

現行FCMSは、Unity版よりも広いHEX盤面、ZOC/ZOC2、同時移動、ロングパスズレ、フリーボール、FK/CK/PKミニゲームを前提にしている。そのため、Unity版を丸ごと置き換えるのは推奨しない。採用すべきなのは、Unity版のうち「サッカーらしい意味付けが明確で、FCMSのHEX設計を壊さない部分」に限る。

| 領域 | 現行FCMS | Unity版 | 判断 |
| --- | --- | --- | --- |
| 通常パスのオフサイド | `src/engine/offside.ts` / `special.ts`。ボール位置制約、ハーフライン制約、グレーゾーンを持つ | 相手2枚目ラインとパス前後位置から確率表で判定 | 現行維持。FCMSの方がサッカールール寄り |
| スペース/スルーパス後のオフサイド | `throughPass` が距離2以内の味方に届くと通常パス同様に `deliveredPass` 扱い。フリーボールを後で拾った時の同チームオフサイドは未実装 | セルへのパス時点では判定せず、味方がフリーボールを拾った時に判定 | Unity版の考え方を採用候補。FCMSの穴を埋められる |
| 通常パスとスペースパスの分離 | UI/エンジン上は `pass` と `throughPass` があるが、テストや処理に「移動後座標へ通常pass」の混在がある | コマへのパス、移動受けパス、セルへのスルーパスが明確 | Unity版の分類思想を採用候補 |
| パスカット | ZOC/ZOC2上のパスカット1、受け手ZOCのパスカット2 | 通過セル、着地セル、GoalAreaのGK 100%カット | 基本は現行維持。GoalArea/GKの特殊カットだけ検討余地 |
| ファウル | タックル成功後、攻撃側サードで25%。PAならPK、PA外ならFK | タックル前にファウル抽選。PA/GAならPK、VA/CrossならFK。通常エリアではタックルへ戻る | 現行維持。Unity版のエリア分類だけ参考 |
| タックル | コスト差、ポジション、ZOC隣接で判定 | 同セル人数、ポジション、コスト表で判定 | 現行維持。HEX/ZOC前提の方がFCMSに合う |
| シュート/GK | シュートコース、ブロック、セーブ、キャッチ、距離を考慮 | GKがコース/GoalAreaにいないとシュート100%。失敗後CK/GK抽選 | 現行維持。Unity版は簡略化が強い |
| FK/CK/PK | ミニゲーム仕様 | コスト表による単発抽選 | 現行維持。Unity版はバランス参考値としてだけ使う |
| 遅延行為 | 明確なエンジン実装なし | 自陣で同一チームが3ターン保持すると遅延 | Unity版の考え方を採用候補 |
| 消極的戦術 | 明確なエンジン実装なし | 9コマ以上が自陣深部、かつボールが外にあるとペナルティ | Unity版の考え方をHEX向けに再定義する価値あり |

### 置き換え候補の優先度

1. スペース/スルーパス後のフリーボール取得時オフサイド  
   Unity版の方が自然。現行FCMSは `throughPass` で誰も即時取得しなかった後、次ターンに味方が拾った時のオフサイド検出がない。

2. パス種別の整理  
   「味方コマへのパス」と「空間へのパス」をエンジン上でも明確に分ける。UI上の理解にも効く。

3. 遅延行為  
   自陣保持が続くと停滞するため、Unity版の3ターン保持ルールをFCMSのターン数・盤面サイズに合わせて調整する。

4. 消極的戦術  
   守備コマを自陣深部へ固める戦術への抑止として有効。ただしUnity版の「9コマ」は小盤面前提なので、FCMSでは自陣ゾーン比率やGK周辺HEX数で再設計する。

### 置き換えない方がよい部分

- シュート判定は現行FCMSの方が盤面、距離、ZOC、GK位置を使えている。
- タックル判定は現行FCMSの方がHEX/ZOCゲームとして筋がよい。
- ファウル判定順は現行FCMS仕様の「タックル成功後にファウルで無効化」の方が、プレイヤーに説明しやすい。
- FK/CK/PKはFCMSのミニゲーム仕様を維持する。Unity版の抽選式へ戻すと、ユーザー操作の見せ場が減る。

結論: Unity版で置き換えるのではなく、`throughPass` / フリーボール / 遅延行為 / 消極的戦術だけをFCMS向けに移植する方針がよい。

### 実装反映メモ（2026-06-23）

上記方針のうち、次をFCMSエンジンへ反映した。

- `throughPass` で空間に出たボールに、最後に触ったチーム・コマ・発生元を保持する。
- `throughPass` 由来のフリーボールを同チームが拾った場合、ターン開始時位置を基準にオフサイド判定する。
- 自陣で同一チームが3ターン保持すると `BATTLE_DELAY` を発生させ、相手GKへボールを移す。
- 9枚以上が自陣深部に固まり、ボールがその外にある場合 `PASSIVE_TACTICS` を発生させる。
- 消極的戦術中のチームは、次ターンのパスがカットされやすくなり、ドリブル時のタックルも受けやすくなる。
